# Wordbook 実装移管仕様書

最終更新: 2026-07-18  
状態: 実装前・初版  
対象読者: 本会話を参照できない実装担当LLM／開発者

## 1. この仕様書だけで最初の認識実験から本体実装まで進める

本プロジェクトは、紙の単語帳では学習が続かない中学3年生に、iPad上で英単語を繰り返し手書きさせる学習アプリである。正しい単語を書くまで次へ進めず、正解時には発音を聞かせる。1プレイの初期目標は100語とし、制限時間を超えると達成条件を厳しくする。

最初から本体を作り込まない。最大の不確実性は手書き英字の認識精度である。最初に1画面の「文字認識ラボ」を作り、実際の筆跡データを集め、自動評価で成立性を確認する。認識品質が合格してからゲーム本体へ進む。

1〜7章は着手前に通読する。8章以降は実装・試験時の参照用である。

## 2. 成功条件は、本人の筆跡を正しく通し、誤字を通さないこと

初版完成の条件は次のとおり。

- iPadとApple Pencilで英単語を手書きできる。
- 正しい筆跡を自動認識し、正解後に英語TTSを再生して次の単語へ進む。
- 誤答・認識失敗・タイムアウト時は同じ単語を再度書かせる。
- Lv1、Lv2、Lv3を選び、各レベルで100語のセッションを実行できる。
- セッション中にアプリを終了しても、同じ状態から再開できる。
- レベル別の成績履歴を端末内へ必ず保存する。
- OCRが正解を認識しない場合に限り、手動正解を使える。使用時は問題情報、OCR結果、筆跡画像を監査記録として保存する。
- 保護者はPINで保護された画面から手動正解記録を確認できる。
- 代表的な端末で100問を完了しても、クラッシュ、履歴欠損、進行不能が起きない。

認識精度の暫定合格基準は、本人から集めた評価データに対して次の値とする。

| 指標 | 暫定基準 |
|---|---:|
| 正しい筆跡の合格率 | 95%以上 |
| スペルミス筆跡の誤合格率 | 1%以下 |
| iPad実機での認識時間 p95 | 500ms以下 |

達成できない場合は本体実装を止め、前処理、認識器、UIまたは手動正解運用を再検討する。数値を都合よく下げて先へ進まない。

## 3. 初版はiPadネイティブとし、Webは筆跡収集とUX試作に限定する

### 3.1 採用技術

| 領域 | 採用 |
|---|---|
| プラットフォーム | iPadOSネイティブアプリ |
| 言語 | Swift |
| UI | SwiftUI |
| 手書き | PencilKit `PKCanvasView`をSwiftUIからラップ |
| 初版の認識 | PencilKitの描画を画像化し、VisionでOCR |
| 読み上げ | AVFoundation `AVSpeechSynthesizer` |
| 構造化データ | SwiftData |
| 筆跡画像 | Application Support配下のファイル |
| テスト | Swift TestingまたはXCTest。プロジェクト内で一方に統一 |

安定版のXcodeとiPadOS SDKを使う。2026-07時点でベータのiPadOS 27専用`PKStrokeRecognizer`は初版の必須機能にしない。ただし、後から認識器だけ交換できる境界を設ける。

推奨Deployment TargetはiPadOS 18.0。実機が対応しない場合は、端末情報を確認して変更する。推測で決めない。

### 3.2 Webで扱ってよい範囲

Web試作を作る場合は、画面寸法、タイマー、スクロール演出、問題難易度、疲労感、筆跡PNGの収集に限定する。ブラウザ用OCRの結果をApple Visionの品質根拠にしてはならない。Web試作のコードを本体へ移植できる前提も置かない。

## 4. アプリは認識、ゲーム進行、保存、表示を分離する

推奨ディレクトリ構成は以下とする。既存プロジェクトの標準構成がある場合は、その規約を優先する。

```text
Wordbook/
  App/
  Domain/
    Models/
    GameEngine/
  Recognition/
    HandwritingRecognizer.swift
    VisionHandwritingRecognizer.swift
    DrawingRenderer.swift
  Speech/
  Persistence/
  Features/
    RecognitionLab/
    Session/
    History/
    ParentAudit/
    Settings/
  Resources/
    words.json
WordbookTests/
  GameEngineTests/
  RecognitionFixtureTests/
  PersistenceTests/
TestFixtures/
  Manifest.json
  Images/
```

UIのViewへ得点計算や出題順を直接書かない。ゲーム進行は副作用の少ない`GameEngine`へ集約し、時刻、乱数、TTS、保存、認識器を注入可能にする。

認識器の最小インターフェースは次を基準にする。

```swift
protocol HandwritingRecognizer: Sendable {
    func recognize(_ input: RecognitionInput) async throws -> RecognitionResult
}

struct RecognitionInput: Sendable {
    let imageData: Data
    let expectedWord: String
}

struct RecognitionResult: Sendable {
    let candidates: [RecognitionCandidate]
    let elapsedMilliseconds: Int
    let engineVersion: String
}
```

将来`PKStrokeRecognizer`を追加しても、`GameEngine`と画面は変更しない。

## 5. ゲーム進行は一つの状態機械として実装する

### 5.1 状態

```text
idle
  → presentingQuestion
  → acceptingInk
  → recognizing
  → correctSpeakingAndTransitioning → presentingQuestion
  → incorrectFeedback → acceptingInk
  → timedOut → acceptingInk
  → manuallyPassed → correctSpeakingAndTransitioning
  → completed
```

同時に複数のOCRを走らせない。新しい筆跡が加わったら古い認識タスクをキャンセルし、古い結果が後から返って画面を進めないよう、問題IDと認識世代IDを照合する。

### 5.2 セッション開始時

- レベルを1つ選ぶ。
- 単語リストから100語を選ぶ。初版は重複なしを原則とする。
- 出題順はシャッフルする。
- `targetCorrectCount = 100`、`netCorrectCount = 0`で開始する。
- 同じセッションを再開できるよう、選択単語、順序、現在位置、マスク、得点、期限を保存する。
- 記録再現のため、シャッフルとLv2マスクに使った乱数seedを保存する。

### 5.3 問題表示とタイマー

- 問題が画面下部へ完全に表示された時点でタイマーを開始する。
- 前問のTTS再生時間を次問の制限時間から差し引かない。
- 残り時間を常時表示し、終盤は色と軽い視覚効果で急かす。強い点滅は使わない。
- 暫定制限時間はLv1=8秒、Lv2=12秒、Lv3=15秒とし、`GameConfiguration`へ集約する。
- セッション中は画面の自動ロックを抑止する。終了・中断時には必ず元へ戻す。

### 5.4 正解時

1. `netCorrectCount`を1増やす。
2. 試行記録を保存する。
3. 英語音声で単語を1回読み上げる。
4. 正解した問題を上へスクロールさせ、次問を下部へ表示する。
5. 次問の表示完了後に新しいタイマーを開始する。
6. `netCorrectCount >= targetCorrectCount`なら完了画面へ進む。

TTSは英語音声を明示して選択する。速度の暫定値は聞き取りやすい低速とし、設定画面で試聴・変更可能にする。速度値は端末で試聴して決め、仕様書の固定小数値を盲目的に使わない。

### 5.5 誤答または認識不能時

- 得点と目標数は変更しない。
- 同じ単語を即座に再出題する。
- 誤りを短く表示した後、キャンバスを消してタイマーを最初から開始する。
- 試行回数とOCR候補は保存する。
- OCRが空結果でも同じ扱いとする。ただし画面文言は「読み取れませんでした」と「つづりが違います」を分ける。

### 5.6 タイムアウト時

- `netCorrectCount = max(0, netCorrectCount - 1)`とする。
- `targetCorrectCount += 1`とする。
- タイムアウト記録を保存する。
- 同じ単語を即座に再出題する。
- キャンバスを消し、タイマーを再開する。

この二重ペナルティは依頼内容をそのまま具体化した暫定決定である。体験が厳しすぎる場合は、実装後にオーナー判断で変更する。実装担当が独断で緩和しない。

アプリがバックグラウンドへ移った時間も期限判定に含める。復帰時に期限を超えていた場合は、タイムアウトを1回だけ適用する。経過時間に比例して複数回罰しない。

### 5.7 手動正解時

ボタン名は「認識されない」とし、「書き直す」ボタンと区別する。確認ダイアログを1回表示する。

押された場合は正解と同じく1点を加算して次へ進む。その前に、問題、表示内容、筆跡画像、OCR候補、信頼度、試行回数、残り時間、日時、レベル、セッションIDを監査記録として保存する。保存に失敗した場合は手動正解を成立させない。

## 6. 難易度は表示情報だけを変え、正解判定は共通化する

### Lv1: そのまま書き写す

英単語と頻出の日本語訳を表示する。正解単語の全文が見える。

### Lv2: 約半数の文字を隠す

英単語、日本語訳、文字数を表示する。文字位置の約50%を`_`などの記号へ置換する。長さ`n`に対して隠す文字数は`max(1, round(n * 0.5))`とする。

- 先頭・末尾を常に見せるという追加ルールは設けない。
- 同一問題の再試行中はマスク位置を変えない。
- seedから同じマスクを再現できるようにする。
- アポストロフィ、ハイフン、空白は隠す対象外とする。

### Lv3: 意味と文字数だけで答える

日本語訳と英字数を表示し、英単語は全て`_`で隠す。ハイフン等を含む語を採用する場合、その扱いを単語データ側で明示する。

## 7. 文字認識ラボで成立性を先に証明する

### 7.1 ラボ画面に必要な機能

- 正解単語の入力または単語リストからの選択
- PencilKitキャンバス
- 「認識」「書き直す」「保存」ボタン
- 自動認識ON/OFF
- OCR上位候補、各confidence、処理時間の表示
- OCRへ渡した実画像のプレビュー
- 画像のcrop範囲、scale、余白、背景色、線色、線幅の表示
- 人間によるラベル付け: 正しい筆跡／意図的な誤字／判別不能
- PNG、`PKDrawing`、JSONメタデータの保存または共有

自動認識は最終ストローク更新から700msのdebounceを暫定値とする。描画変更のたびに実行中タスクをキャンセルする。

### 7.2 Vision OCRの基準設定

- 認識言語は英語を優先する。
- 初期値はaccuracy優先とする。
- 上位3候補を取得する。
- baselineでは正解単語を`customWords`へ入れない。答えへの強い誘導で誤字を通す危険があるためである。
- 比較前にUnicode正規化、前後空白削除、小文字化を行う。
- アルファベット以外の除去は、単語データのアポストロフィ・ハイフン方針と一致させる。
- Levenshtein距離による類似一致は初版で正解にしない。分析値としてだけ保存する。
- 候補のいずれかが正規化後の正解と完全一致し、必要なconfidence閾値を満たしたときだけ合格とする。

confidence閾値は固定値を先に決めない。収集データを用い、正解合格率と誤合格率の両方を見て選ぶ。

### 7.3 画像前処理は比較実験できる形にする

少なくとも次を設定化し、同じfixtureへ一括適用できるようにする。

- 筆跡bounds周囲の余白
- 出力画像scale
- 背景色と筆跡色
- grayscale化
- contrast調整
- 二値化の有無

各方式にバージョン名を付け、評価結果に保存する。最良の1例だけで判断せず、全fixtureの集計で選ぶ。

## 8. 一度集めた筆跡は固定fixtureとして何度でも使う

本人に20〜50語を書いてもらい、各語で丁寧、急ぎ、小さい、大きい、つながった、書き直しありの筆跡を集める。さらに、1文字欠落、余分な文字、文字順違い、似た文字への置換を意図的に作る。

メタデータ例:

```json
{
  "id": "beautiful-correct-fast-001",
  "expectedWord": "beautiful",
  "label": "correct",
  "writerAlias": "student-a",
  "inputStyle": "fast",
  "imageFile": "Images/beautiful-correct-fast-001.png",
  "drawingFile": "Drawings/beautiful-correct-fast-001.pkdrawing",
  "createdAt": "2026-07-18T10:00:00+09:00"
}
```

実名、年齢、学校名などは保存しない。Gitへ入れる場合は、筆跡が個人データになり得ることを踏まえ、明示的な許可を得る。

評価コマンドまたはテストは、fixture全件について候補、confidence、処理時間、正誤をCSVまたはJSONへ出力し、成功条件の3指標を集計する。認識処理を変えたら同じfixtureで回帰試験する。

## 9. 保存データは再開、成績、監査の三用途に分ける

### 9.1 単語データ

```json
{
  "id": "word-0001",
  "word": "apple",
  "meaningJa": "りんご",
  "enabled": true,
  "source": "owner-provided",
  "tags": ["junior-high"]
}
```

単語と訳の出典はオーナーが用意する。著作権のある単語帳から無断転記しない。同一英単語に複数の訳を持たせる場合も、出題時は指定された「頻出の意味」を1つ表示する。

### 9.2 SwiftDataモデル

- `PlaySession`: ID、level、開始・終了時刻、状態、seed、目標数、正答数、現在問題、選択単語順、期限
- `Attempt`: sessionID、wordID、結果種別、開始・終了時刻、OCR候補、処理時間、試行番号
- `ScoreHistory`: level、完了時刻、所要時間、最終目標数、正答数、timeout数、誤答数、手動正解数
- `ManualPassEvidence`: sessionID、attemptID、wordID、表示問題、画像相対パス、OCR情報、日時、確認状態
- `AppSettings`: レベル別制限時間、TTS速度、保護者PINのハッシュとsalt

画像本体をSwiftDataへ大量に埋め込まない。Application Support配下へUUID名で保存し、モデルには相対パスを保存する。レコード保存とファイル保存の片方だけが成功しないよう、失敗時の補償処理を用意する。

### 9.3 保護者画面

- PIN認証後だけ開ける。
- 手動正解を未確認優先で一覧表示する。
- 単語、意味、レベル、日時、筆跡画像、OCR候補を表示する。
- 「確認済み」「要注意」の状態を付けられる。
- 学習者側から記録を削除できない。
- 初版は端末内保存のみ。アプリ削除で失われることを設定画面に明記する。

## 10. 実装は認識リスクから順に進める

### Phase 0: 開発環境を確認する

確認項目はMac機種、macOS、Xcode、対象iPad機種、iPadOS、Apple Pencil、Apple Account、配布方法である。自動署名を使い、最初の実機起動と再起動を確認する。

完了条件: 空のSwiftUIアプリをSimulatorとiPad実機の両方で起動できる。

### Phase 1: 文字認識ラボを作る

PencilKit入力、画像化、Vision OCR、候補表示、fixture保存を実装する。本体のゲーム画面はまだ作らない。

完了条件: 同じ筆跡を保存・再読込でき、OCR候補と時間を確認できる。

### Phase 2: 筆跡を収集して自動評価する

正解と誤字のfixtureを集め、前処理と閾値を比較する。

完了条件: 2章の認識指標を自動集計でき、採用設定と評価結果を文書へ残せる。基準未達ならPhase 1へ戻る。

### Phase 3: 純粋なゲームエンジンを作る

UIなしで、出題、正解、誤答、timeout、目標増加、完了判定、再開状態を実装する。

完了条件: 境界ケースを含む単体テストが通る。

### Phase 4: Lv1を垂直に完成させる

100語選択、手書き、認識、TTS、スクロール、保存、再開、完了履歴を接続する。

完了条件: 実機で短縮10問テストと通常100問テストが完走する。

### Phase 5: Lv2、Lv3と保護者監査を追加する

マスク生成、意味のみ出題、手動正解証跡、PIN画面を追加する。

完了条件: 各レベルの履歴が分離され、手動正解画像を保護者画面で確認できる。

### Phase 6: 実利用で調整する

本人によるセッションで時間、OCR失敗、誤合格、疲労、手動正解率を測る。調整値と理由を記録する。

完了条件: 3回以上の100問セッションで進行不能がなく、手動正解率と誤合格率が許容範囲に入る。

## 11. テストはゲーム規則とデータ喪失を重点的に守る

最低限、次を自動化する。

- 正解で正答数が1増え、次問へ進む。
- 誤答で得点が変わらず、同じ問題になる。
- timeoutで正答数が下限0を守って1減り、目標が1増える。
- 最終問題のtimeout後も早期完了しない。
- 手動正解の証跡保存失敗時は得点を増やさない。
- Lv2のマスクは約半数で、同じseedと問題なら再現する。
- Lv3では正解文字を表示しない。
- 古いOCR応答が次の問題を正解にしない。
- 連続stroke更新でOCRが並行実行されない。
- バックグラウンド復帰時のtimeoutは1回だけ適用される。
- 中断したセッションが問題、マスク、得点、目標を保って再開する。
- レベル別履歴が混ざらない。
- 画像と監査レコードの不整合を検出・回復できる。
- 100問にtimeoutと誤答を混ぜても完了条件が正しい。

実機確認ではApple Pencilの追従、誤タッチ、画面回転方針、TTS音量、100問後のメモリ使用量を確認する。初版は横画面固定を推奨するが、対象者の持ち方を見て確定する。

## 12. 実装担当LLMは変更を小さく保ち、各Phaseで止まって証拠を残す

- 一度に複数Phaseを実装しない。
- 各Phase開始時に対象ファイルと検証方法を短く提示する。
- APIの存在やOS要件は、その時点のApple公式ドキュメントで確認する。
- ベータAPI、外部OCR SDK、クラウド、アカウント機能、同期、課金を無断で追加しない。
- OCR精度不足を手動正解ボタンの多用で隠さない。
- 単語データの仮データと正式データを混同しない。
- 仕様変更が必要なら、理由、影響範囲、推奨案を示してオーナーの承認を待つ。
- 実装後は実行したテスト、未実行の実機確認、既知の制約を記録する。

## 13. 着手前にオーナーへ確認する項目

次は未確定であり、Phase 0でまとめて確認する。回答がなくても文字認識ラボまでは仮値で進めてよいが、本体リリース前には確定が必要である。

1. 開発に使えるMacの機種、macOS、空き容量
2. 対象iPadの機種、iPadOS、Apple Pencilの型
3. 家庭内利用、TestFlight、App Storeのどれを想定するか
4. 単語100語以上と頻出の日本語訳の入手元
5. 横画面固定でよいか
6. 制限時間の初期値
7. timeoutの二重ペナルティを意図どおり採用するか
8. 保護者PINを忘れた場合の復旧方法
9. 監査画像の保持期間と削除権限
10. 手動正解を得点上の通常正解として扱うか

## 14. 用語集

- **文字認識ラボ**: 本体より先に作る、筆跡取得とOCR評価だけの1画面アプリ。
- **fixture**: 認識処理を何度変更しても同じ条件で再試験するための固定筆跡データ。
- **OCR候補**: 認識器が返す文字列候補。1位だけでなく上位候補を評価する。
- **confidence**: OCR候補に対する認識器の確からしさ。値だけで正解を保証しない。
- **debounce**: 書いている途中の連続実行を避け、最後の入力から短時間待ってOCRを1回実行する制御。
- **GameEngine**: UIから独立して、得点、timeout、出題順、完了条件を管理する部分。
- **手動正解**: OCR失敗時に限って人間が進行を許可する操作。必ず監査証跡を残す。

この仕様書で解決しない判断は、推測で実装せず、13章の確認項目へ追記してオーナーへ提示する。
